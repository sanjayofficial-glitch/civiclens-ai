# Bugfix Requirements Document

## Introduction

BlockSeBlock is an AI-powered civic issue reporting platform being stabilized for a hackathon demo. Five interconnected bugs currently prevent the "golden demo" flow from working end-to-end: the AI image analysis pipeline fails silently, the reporting pipeline has broken links that prevent newly submitted reports from appearing across the app, location detection returns the wrong city, uploaded images do not render after a page reload, and several views may still rely on mock data instead of live Firebase reads. All five must be fixed so the following demo sequence works reliably: sign in → upload photo → AI analyzes it → submit report → see it in My Reports, Map, and Dashboard immediately → verify image, AI analysis, and location in the detail view.

---

## Bug Analysis

### Current Behavior (Defect)

**Bug 1 — AI Image Analysis**

1.1 WHEN the `analyzeIssueImage` callable is invoked and `GEMINI_API_KEY` is absent from the Cloud Functions runtime environment THEN the system throws a `failed-precondition` HttpsError and the callable returns an error response instead of an analysis result

1.2 WHEN `analyzeIssueMedia` in `geminiService.ts` calls the Gemini REST endpoint and the Firebase Storage download URL passed as `imageUrl` is a `googleapis.com/download/storage/v1/b/…` signed URL THEN the system fetches the image as a binary buffer and re-encodes it as base64, but the `fetchFileBuffer` call in `storageService.ts` may fail with a non-200 status because the URL is not publicly accessible, causing `imageParts` to be empty and Gemini to analyze text-only

1.3 WHEN Gemini returns a valid JSON response but with field names `title` and `description` (as generated by `geminiService.ts`) instead of `suggestedTitle` and `suggestedDescription` (as expected by the callable's `CallableResponse` type in `ai.service.ts`) THEN the system maps the analysis correctly on the server but the client-side `AiService.analyzeViaCallable` reads `a.title` and `a.description` and returns an `IssueAiAnalysis` with `suggestedTitle` populated from `a.title` — this mapping works, but if the callable's `analysis` object is missing `title`/`description` the `suggestedTitle`/`suggestedDescription` fields are `undefined`

1.4 WHEN the Cloud Function `onIssueCreated` trigger fires after issue creation and `enrichIssueOnCreate` calls `analyzeIssueMedia` with `imageUrls` from `issue.media.images` THEN the system passes Firebase Storage download URLs directly to `fetchFileBuffer`, but the default Storage bucket ACL may reject unauthenticated HTTP GET requests, causing the fetch to fail and Gemini to produce a low-confidence text-only result

1.5 WHEN `analyzeIssueMedia` exhausts all retry attempts and all retries throw THEN the system falls back to `fallbackAnalysis`, which correctly returns a result but sets `confidence` to `0.35` (no images) or `0.55` (images present) — however callers receive no signal that the AI path failed, making the fallback indistinguishable from a real Gemini result

**Bug 2 — Reporting Pipeline**

2.1 WHEN the ReportWizard uploads photos via `UploadService.uploadFile` with path pattern `issues/${Date.now()}_${idx}_${random}` (no leading folder slash, no `users/` prefix) THEN the system writes to a storage path that does not match the `storage.rules` pattern `users/{userId}/issue/{allPaths=**}`, causing the upload to be rejected with `storage/unauthorized` if rules are enforced, meaning `successfulUrls` is empty and the issue is saved with `media.images: []`

2.2 WHEN `IssueService.create` saves the issue document with `media.images: []` (upload failed or no URLs) THEN the system creates the issue without any image URL, so `onIssueCreated` triggers `enrichIssueOnCreate` with an empty `imageUrls` array, and Gemini performs text-only analysis

2.3 WHEN a new issue is submitted and `IssueService.create` returns a `docRef`, the wizard calls `navigate(\`/issues/${docRef.id}\`)` THEN the system navigates to the detail page but `useIssues` on the Home feed, Map page, and Government Dashboard still show the cached list from before submission because `listenToIssues` uses a real-time `onSnapshot` listener — if the Firestore composite index for `createdAt desc` is not yet built, the listener falls back to a one-shot `getDocs` query and will not pick up newly written documents until the page is manually refreshed

2.4 WHEN `onIssueCreated` trigger fires and `enrichIssueOnCreate` completes the AI analysis, it writes `aiAnalysis` back to the issue document via `snap.ref.set({ aiAnalysis, ... }, { merge: true })` THEN the system updates the document but the `IssueDetailsPage` uses `useIssue` which calls `listenToIssue` — if the real-time listener is active the UI updates; however, if the user navigated directly to the detail page before the trigger completed, the initial render shows no `aiAnalysis` section until the listener fires again

**Bug 3 — Location Accuracy**

3.1 WHEN the ReportWizard enters Step 2 (location) and `draft.hasCustomLocation` is `false`, the `useEffect` fires `GeolocationService.getCurrentPosition()` THEN the system calls Stage 1 (`enableHighAccuracy: false, maximumAge: 30000`) which may return a cached network/cell-tower position that places the user in Kolkata (or any other city) instead of Rourkela, because `maximumAge: 30000` allows the browser to return a stale fix up to 30 seconds old, and the stale fix reflects a previously cached IP-geolocation position that was never corrected

3.2 WHEN the browser returns a coarse position (Stage 1 success) THEN the system sets `resolved = true` immediately and never waits for the Stage 2 high-accuracy GPS refinement, so the inaccurate Stage 1 result is committed to `draft.latitude`/`draft.longitude` and displayed on the map

3.3 WHEN the reverse-geocode call to Nominatim completes for the wrong coordinates THEN the system sets `draft.address` to the city/locality associated with those wrong coordinates (e.g., "Kolkata, West Bengal"), and this address is persisted into the issue document's `location.address` field

3.4 WHEN `draft.hasCustomLocation` is already `true` (user has previously set a location) and the user navigates back to step 2 THEN the system re-runs the auto-detection `useEffect` because the dependency array is `[draft.step]` only, not `[draft.step, draft.hasCustomLocation]`, potentially overwriting a manually chosen location

**Bug 4 — Image Display**

4.1 WHEN a report is submitted successfully and the user navigates to `IssueDetailsPage` immediately after creation THEN the system displays the image from `issue.media.images[0]` via the `<img src={images[imageIndex]}>` tag; if the Storage URL is a Firebase download URL with a token (`?alt=media&token=...`) the image loads correctly, but if the token has expired or the URL is a raw `gs://` path that was incorrectly stored in Firestore THEN the `<img>` element renders a broken image placeholder

4.2 WHEN `IssueCard` renders in "horizontal" or "default" variant and reads `issue.media?.thumbnail ?? issue.media?.images?.[0]` THEN the system uses the first image URL as the thumbnail; if `media.images` is an empty array (upload failed, see 2.1) the thumbnail is `undefined` and the card shows a category icon instead of a photo, even when the user clearly uploaded an image

4.3 WHEN the user reloads the page after submitting a report THEN the system re-fetches the issue from Firestore; if the `media.images` array was saved with `blob:` URLs (local object URLs that are revoked after navigation) instead of `https://firebasestorage.googleapis.com` URLs THEN the `<img>` elements display broken images because `blob:` URLs are not persistent across page loads

4.4 WHEN `IssueService.getIssues` or `listenToIssues` returns issues from Firestore and the `issueConverter` in `converters.ts` deserializes the document THEN if the Firestore document stores `media.images` as a subcollection or with non-standard field names, the converter may return `issue.media` as `undefined`, causing `IssueCard` to skip image rendering entirely

**Bug 5 — Mock Data Remaining**

5.1 WHEN `HomePage` renders the "Nearby Issues" section THEN the system queries `useIssues({}, 10)` which calls `IssueService.listenToIssues` — this is live data; however, if no Firestore composite index exists for `createdAt desc` the listener fires an error, falls back to a one-shot query with no `orderBy`, and may return stale or empty results that look like mock data

5.2 WHEN `HomePage` renders the "Community Impact" stats section THEN the system calls `useCommunityStats` which listens to `analytics/global` in Firestore; if the `analytics/global` document has never been written (no issue has been resolved or the `onIssueUpdated` trigger has not yet run) THEN `stats` is `null` and the UI shows `-` for total reports and `0` for resolved this week, which during a demo looks like hardcoded zeroes rather than real counts

5.3 WHEN `GovernmentDashboardPage` renders stat cards THEN the system computes `activeIssues` and `totalReports` as fallbacks from the live `issues` array when `stats` is `null`; the "Resolved This Week" stat falls back to `0` and "Total Verifications" falls back to `0` regardless of actual data because the `analytics/global` document may not exist

5.4 WHEN `LeaderboardPage` renders THEN the system queries the `leaderboard` collection via `LeaderboardService.listenToLeaderboard(period, 50)` which is live Firebase data; however, the leaderboard collection is only populated by the `updateLeaderboard` callable (admin/moderator only) and the `onAuthUserCreated` trigger — if neither has run for newly created users since the last report submission THEN the leaderboard shows empty or stale entries that do not reflect the current reporter's score

5.5 WHEN `NotificationsPage` renders THEN the system queries `useNotifications` which calls `NotificationService.listenToNotifications` — this is live Firebase data; if the user has no notification documents yet the page shows "No notifications" correctly, but the `useNotifications` hook may query with `userId` undefined on first render (before auth resolves) and cache an empty result that is not re-fetched after auth completes

### Expected Behavior (Correct)

**Bug 1 — AI Image Analysis**

2.1 WHEN the `analyzeIssueImage` callable is invoked and `GEMINI_API_KEY` is absent from the runtime THEN the system SHALL log a structured error, return a fallback analysis result with `confidence: 0.35` and `category: 'other'` rather than throwing an unhandled error, and the callable SHALL return `{ status: 'fallback', analysis: { ... } }` so the wizard can display it

2.2 WHEN `analyzeIssueMedia` is called with a Firebase Storage download URL THEN the system SHALL fetch the image using an authenticated Admin SDK method (e.g., `bucket.file(path).download()`) or convert the `https` URL to a storage path and use `admin.storage()` directly, so the fetch succeeds regardless of public access settings

2.3 WHEN the Gemini REST API returns a JSON object with `title` and `description` fields THEN the system SHALL map them to `suggestedTitle` and `suggestedDescription` consistently in both the Cloud Function response and the client-side `CallableResponse` type so no fields are ever `undefined`

2.4 WHEN `onIssueCreated` triggers `enrichIssueOnCreate` and the issue has images in `media.images` THEN the system SHALL successfully download and base64-encode at least the first image using the Admin SDK, pass it to Gemini as an `inlineData` part, and return an `aiAnalysis` with `confidence >= 0.6` for a real photo

2.5 WHEN all Gemini retry attempts fail THEN the system SHALL use `fallbackAnalysis` and write it to the issue document with a flag `aiAnalysis.usedFallback: true` so callers can distinguish fallback results from real AI results

**Bug 2 — Reporting Pipeline**

2.6 WHEN the ReportWizard uploads a photo via `UploadService.uploadFile` THEN the system SHALL use a storage path that includes the authenticated user's UID (e.g., `users/${userId}/issue/${timestamp}_${random}_photo.jpg`) so the upload matches the Firebase Storage security rules and succeeds

2.7 WHEN the upload completes successfully THEN the system SHALL receive a `https://firebasestorage.googleapis.com` download URL, store it in `draft.photos`, and include it in `media.images` when the issue document is saved to Firestore

2.8 WHEN `IssueService.create` saves the issue document THEN the system SHALL immediately return the new document reference, and the real-time `onSnapshot` listeners in `useIssues` (Home feed), `useIssues` (Map page), and `useIssues` (Dashboard) SHALL reflect the new issue within 2 seconds without requiring a manual page refresh

2.9 WHEN `enrichIssueOnCreate` completes and writes `aiAnalysis` back to the issue document THEN the system SHALL have the real-time listener in `IssueDetailsPage` pick up the update and re-render the AI Analysis section without a page reload

**Bug 3 — Location Accuracy**

3.1 WHEN the ReportWizard auto-detects location on entering Step 2 THEN the system SHALL request high-accuracy GPS from the browser first (Stage 1 with `enableHighAccuracy: true, maximumAge: 0`) and SHALL NOT accept a stale cached position; if the GPS fix takes more than 10 seconds the system SHALL fall back to a low-accuracy network fix

3.2 WHEN a GPS position is obtained THEN the system SHALL set `draft.latitude` and `draft.longitude` to the coordinates reported by the browser, and the reverse-geocode result SHALL correctly identify the city as the one matching those coordinates

3.3 WHEN `draft.hasCustomLocation` is already `true` on entering Step 2 THEN the system SHALL NOT overwrite the existing coordinates with a new auto-detected position

3.4 WHEN the issue document is saved THEN `location.address` SHALL contain the city and locality that match the stored `location.geopoint` coordinates, and both SHALL refer to the same physical location (Rourkela when the device is in Rourkela)

**Bug 4 — Image Display**

4.1 WHEN a report is submitted and the user navigates to `IssueDetailsPage` THEN the system SHALL display the uploaded image immediately using the `https://firebasestorage.googleapis.com` download URL stored in `media.images[0]`, and the image SHALL remain visible after a full page reload

4.2 WHEN `IssueCard` renders in any variant THEN if `media.images` contains at least one valid `https://` URL the system SHALL display it as the thumbnail; only if `media.images` is empty SHALL the system render the category icon fallback

4.3 WHEN `IssueService.create` saves the issue THEN `media.images` SHALL contain only `https://firebasestorage.googleapis.com` URLs; the system SHALL never persist `blob:` object URLs to Firestore

4.4 WHEN the `issueConverter` deserializes an issue document from Firestore THEN it SHALL correctly map `media.images`, `media.videos`, and `media.thumbnail` to the typed `Issue` object so that downstream components always receive a well-formed `media` object

**Bug 5 — Mock Data Remaining**

5.1 WHEN `HomePage` mounts and the Firestore composite index for `createdAt desc` is available THEN the system SHALL display real issues from the `issues` collection sorted newest-first within 2 seconds of page load; newly submitted reports SHALL appear in the feed without a refresh

5.2 WHEN a new issue is created, the `analytics/global` document SHALL be updated (either by the `onIssueCreated` trigger or by a client-side write after submission) so that `totalReports` increments immediately and Community Impact stats on the Home page show a non-zero count

5.3 WHEN `GovernmentDashboardPage` renders and `analytics/global` exists THEN the system SHALL display real counts for Active Issues, Resolved This Week, Total Reports, and Total Verifications from that document; the fallback computation from the live `issues` array SHALL be used only when the analytics document is absent

5.4 WHEN a user submits a new issue and the `updateLeaderboard` callable has not been run manually THEN the system SHALL still show the user's initial leaderboard entry (created by `onAuthUserCreated`) with their current reputation score, so the Leaderboard page is not empty for newly registered demo users

5.5 WHEN `useNotifications` mounts THEN the system SHALL wait for `user?.uid` to be defined before starting the Firestore listener, and SHALL re-subscribe when the UID becomes available after auth resolution, so notifications are not permanently empty after a cold-start

---

### Unchanged Behavior (Regression Prevention)

3.1 WHEN a user manually clicks a location on the map in Step 2 of the ReportWizard THEN the system SHALL CONTINUE TO set the marker and coordinates to the clicked point and SHALL NOT overwrite them with auto-detected GPS

3.2 WHEN a user types an address in the address search field and Nominatim returns a result THEN the system SHALL CONTINUE TO update the map center and coordinates to the geocoded position

3.3 WHEN the Gemini API is reachable and `GEMINI_API_KEY` is configured THEN the system SHALL CONTINUE TO call the real Gemini model and return AI-generated `category`, `severity`, `confidence`, `suggestedTitle`, `suggestedDescription`, and `suggestedTags`

3.4 WHEN `IssueService.getIssues` is called with filter constraints (`status`, `category`, `severity`, `reporterId`) THEN the system SHALL CONTINUE TO apply those filters in the Firestore query and return only matching documents

3.5 WHEN a user submits a report and the `onIssueCreated` trigger fires THEN the system SHALL CONTINUE TO award reputation points to the reporter and run duplicate detection alongside the AI analysis

3.6 WHEN `VoteService.castVote` is called THEN the system SHALL CONTINUE TO perform the Firestore transaction atomically (unvote, switch, and new-vote paths all intact)

3.7 WHEN `IssueDetailsPage` renders an issue with no images THEN the system SHALL CONTINUE TO skip the image gallery section and show only the title, description, badges, voting, and map sections

3.8 WHEN the `analyzeIssueImage` callable is invoked with a direct `imageUrl` and the server-side path succeeds THEN the system SHALL CONTINUE TO return `{ status: 'success', analysis: { ... } }` with the same response shape expected by `AiService.analyzeViaCallable`

3.9 WHEN `IssueService.listenToIssues` encounters a `failed-precondition` error (missing index) THEN the system SHALL CONTINUE TO fall back to a simple equality-filter one-shot query with client-side sort rather than returning an error

3.10 WHEN the user is on any page that uses `useIssues`, `useUser`, `useComments`, or `useNotifications` and the Firestore connection is live THEN the system SHALL CONTINUE TO update the UI in real-time when documents are written or updated by other clients or Cloud Functions

## Bug Condition Pseudocode

### Bug 1 — AI Image Analysis

```pascal
FUNCTION isBugCondition_AI(X)
  INPUT: X = { apiKeyPresent: bool, imageUrl: string, storageAccessible: bool }
  OUTPUT: boolean
  RETURN NOT X.apiKeyPresent
      OR NOT X.storageAccessible
END FUNCTION

// Property: Fix Checking
FOR ALL X WHERE isBugCondition_AI(X) DO
  result ← analyzeIssueImage'(X)
  ASSERT result.status IN ['success', 'fallback']
  ASSERT result.analysis.category IS NOT NULL
  ASSERT result.analysis.severity IS NOT NULL
  ASSERT result.analysis.confidence IS NUMBER
END FOR

// Property: Preservation Checking
FOR ALL X WHERE NOT isBugCondition_AI(X) DO
  ASSERT analyzeIssueImage(X) = analyzeIssueImage'(X)
END FOR
```

### Bug 2 — Reporting Pipeline

```pascal
FUNCTION isBugCondition_Pipeline(X)
  INPUT: X = { uploadPath: string, userUID: string }
  OUTPUT: boolean
  RETURN NOT X.uploadPath.startsWith('users/' + X.userUID + '/')
END FUNCTION

// Property: Fix Checking
FOR ALL X WHERE isBugCondition_Pipeline(X) DO
  result ← submitReport'(X)
  ASSERT result.mediaImages.length > 0
  ASSERT ALL url IN result.mediaImages: url.startsWith('https://')
END FOR
```

### Bug 3 — Location Accuracy

```pascal
FUNCTION isBugCondition_Location(X)
  INPUT: X = { cachedPositionAge_ms: number, enableHighAccuracy: bool }
  OUTPUT: boolean
  RETURN X.cachedPositionAge_ms > 0 AND NOT X.enableHighAccuracy
END FUNCTION

// Property: Fix Checking
FOR ALL X WHERE isBugCondition_Location(X) DO
  result ← getCurrentPosition'(X)
  ASSERT result.coords.accuracy < 100  // metres
  ASSERT reverseGeocode(result.coords) CONTAINS actualCityName
END FOR
```

### Bug 4 — Image Display

```pascal
FUNCTION isBugCondition_Image(X)
  INPUT: X = { storedUrl: string }
  OUTPUT: boolean
  RETURN X.storedUrl.startsWith('blob:') OR X.storedUrl = ''
END FUNCTION

// Property: Fix Checking
FOR ALL X WHERE isBugCondition_Image(X) DO
  result ← getIssueMedia'(X)
  ASSERT ALL url IN result.images: url.startsWith('https://firebasestorage.googleapis.com')
END FOR
```

### Bug 5 — Mock Data

```pascal
FUNCTION isBugCondition_MockData(X)
  INPUT: X = { analyticsDocExists: bool, leaderboardInitialized: bool, userUidResolvedBeforeQuery: bool }
  OUTPUT: boolean
  RETURN NOT X.analyticsDocExists
      OR NOT X.leaderboardInitialized
      OR NOT X.userUidResolvedBeforeQuery
END FUNCTION

// Property: Preservation Checking — all pages must source data from Firestore
FOR ALL views IN [Home, MyReports, Map, Dashboard, Notifications, Leaderboard] DO
  ASSERT view.dataSource = 'firestore'
  ASSERT view.mockDataUsed = false
END FOR
```
